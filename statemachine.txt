⏺ Reference saved. Here's what the fprime-sensors ImuManager reveals and how it
  should inform our HS2 docs:

  ---
  Key findings from fprime-sensors ImuManager

  1. Hardware managers are queued, not active

  The ImuManager is a queued component — it has a message queue but no dedicated
   thread. It runs entirely on the rate group thread that calls it. Our HS2 SDD
  currently says all hardware managers are Active. For managers that are purely
  rate-group-driven with no need for an independent thread, queued is the
  correct type and uses fewer system resources.

  Suggestion: Reconsider hardware manager component types. Managers like
  ImuManager, SunSensorManager, TemperatureSensorManager that only respond to
  rate group ticks are good candidates for queued. Managers that need to run
  independently of rate groups (e.g., EnduroSatManager handling incoming radio
  bytes asynchronously) would stay active.

  ---
  2. Startup states should be named, not labeled [Startup]

  The fprime-sensors startup sequence is explicit:

  RESET → WAIT_RESET → ENABLE → CONFIGURE → RUN

  Every state on error returns to RESET (self-healing). There's no persistent
  Error state — the machine always tries to recover. Our SDD placeholder
  [Startup] should be replaced with real named states derived from the device's
  initialization sequence.

  Suggestion: Update all hardware manager state machines from [Startup] → Idle →
   Sampling → Idle / Error to explicit named states. Error is not a resting
  state — on error the machine resets and retries.

  ---
  3. Hardware bus ports belong on the manager directly

  The ImuManager has two output ports declared directly on itself:

  output port busWrite: Drv.I2c
  output port busWriteRead: Drv.I2cWriteRead

  These connect at the topology level to a LinuxI2cDriver instance. There's no
  abstraction layer — the manager IS the hardware interface. Our port tables
  don't list these bus ports at all.

  Suggestion: Add bus interface ports to each hardware manager's port table. For
   example, ImuManager needs busWrite: Drv.I2c and busWriteRead: 
  Drv.I2cWriteRead. UART-based managers (StarTrackerManager, GnssManager,
  EnduroSatManager) would use Drv.ByteStreamSend / Drv.ByteStreamRecv instead.

  ---
  4. Parameters drive hardware configuration, not constructor args

  Range settings (accelerometer/gyroscope scale) are F' param values.
  parameterUpdated() is called by the framework when ground updates a parameter
  — it sends a reconfigure signal, transitioning from RUN directly back to
  CONFIGURE without a full reset.

  Suggestion: Hardware configuration that operators may want to adjust in flight
   (e.g., camera exposure, IMU range) should be modeled as F' parameters, not
  just commands. This pattern is already used in our SDD via prmGet ports but
  the reconfigure signal pattern is worth calling out explicitly.

  ---
  One question before I propose doc changes

  The queued vs active distinction is the most architecturally significant
  finding. Do you want to adopt queued for the rate-group-driven hardware
  managers (IMU, sun sensors, temperature sensors, cameras), keeping active only
   for managers that need an independent thread?

   where this happens:

  Level 1 — SatStateMachine → Application Manager

   AdcsManager to enter Precision Pointing, or tells DataCollectionManager to
  start/stop an experiment). This is already in the SDD.

  Level 2 — Application Manager → Hardware Manager
  The application manager is responsible for powering its hardware managers on
  and off as part of its own workflow. This is already modeled in
  DataCollectionManager — it has explicit cameraPowerOn[2] and cameraPowerOff[2]

  The hardware manager's state machine owns this — it would have an explicit
  POWERED_OFF state as its initial state, only transitioning to RESET → ENABLE →
   CONFIGURE → RUN when the application manager sends a power-on signal. That's
  where the device actually gets powered up and initialized.

  So the pattern is:
  SatStateMachine ──mode cmd──▶ DataCollectionManager ──powerOn/Off──▶

  Approach 1: Single SM, mode as a signal

  One SM per application component. All modes are states within the same SM.
  modeIn_handler maps the incoming mode to the appropriate switch signal.

  AdcsApplication — most complex case:

      signal tick
      signal error
      signal switchOff
      signal switchDetumble
      signal switchSunPointing
      signal switchAntennaPointing
      signal switchEarthLimb
      signal switchAttitudeHold

      initial enter OFF

      state OFF {
          on switchDetumble       enter DETUMBLE
          on switchSunPointing    enter SUN_POINTING
          on switchAntennaPointing enter ANTENNA_POINTING
          on switchEarthLimb      enter EARTH_LIMB
          on switchAttitudeHold   enter ATTITUDE_HOLD
      }

      state DETUMBLE {
          on tick               do { runBdot }
          on switchSunPointing  enter SUN_POINTING
          on switchOff          enter OFF
          on error              enter OFF
      }

      state SUN_POINTING {
          on tick              do { runSunPoint }
          on switchDetumble    enter DETUMBLE
          on switchOff         enter OFF
          on error             enter OFF

      # ... remaining modes follow same pattern
  }

  states in the same SM:

  ---

  One SM per component. Mode is the top-level state. Each mode contains nested
  operational substates. Mode-switch signals are handled at the top level and
  are valid from any nested state.

  AdcsApplication:
  state machine AdcsSM {
      signal tick
      signal error
      signal switchMode: AdcsMode   # carries the target mode as data

      initial enter OFF

      state OFF { }

      state DETUMBLE {
          state RUNNING {
              on tick  do { runBdot }
              on error enter DETUMBLE   # resets to DETUMBLE entry
          }
      }

      state SUN_POINTING {
          state ACQUIRING {
              on tick    do { acquireSun }
              on success enter SUN_POINTING.TRACKING
          }
          state TRACKING {
              on tick  do { trackSun }
              on error enter SUN_POINTING.ACQUIRING
          }
      }

          state ACQUIRING { ... }
          state TRACKING  { ... }
      }

          state ACQUIRING { ... }
          state TRACKING  { ... }
      }

  From nasa/fpp — docs/users-guide/Defining-State-Machines.adoc:

  ▎ "When a transition is defined in a parent state, it behaves as if it were 
  ▎ defined in each leaf state that is a descendant."

  And substates to any depth:

  ▎ "Substates may themselves have substates to any depth. All final transition 
  ▎ targets must reach a leaf state."

  This directly enables the mode-switching pattern. The switchMode signal
  defined once at the top level is inherited by every nested leaf state:

  state machine AdcsSM {
      signal switchMode: AdcsMode
      signal tick
      signal error

      initial enter OFF

      state OFF { }

      state DETUMBLE {
          initial enter RUNNING
          state RUNNING { on tick do { runBdot } }
      }

      state SUN_POINTING {
          initial enter ACQUIRING
          state ACQUIRING {
              on tick    do { acquireSun }
              on success enter TRACKING
          }
          state TRACKING {
              on tick do { trackSun }
          }
      }

      # Defined once at top level — inherited by DETUMBLE.RUNNING,
      # SUN_POINTING.ACQUIRING, SUN_POINTING.TRACKING, etc.
      on switchMode(AdcsMode.Off)          enter OFF
      on switchMode(AdcsMode.Detumble)     enter DETUMBLE
      on switchMode(AdcsMode.SunPointing)  enter SUN_POINTING
  }

  The docs also confirm that entry and exit actions across state boundaries
  follow the least common ancestor rule automatically — so switching from
  SUN_POINTING.TRACKING to DETUMBLE will correctly execute exit actions up
  through SUN_POINTING, then entry actions down into DETUMBLE.RUNNING. You get
  clean mode transition behavior for free.

  The one constraint to note: each parent state requires exactly one initial 
  specifier — so SUN_POINTING must declare initial enter ACQUIRING. This means
  every mode switch into SUN_POINTING always starts from ACQUIRING, not from
  wherever it last left off. That's the correct behavior here — a mode re-entry
  should restart from the beginning.
